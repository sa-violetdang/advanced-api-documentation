Error format
GET/error

This document describes standard error responses for reuse.

Response processing

  • [Response.error.code]: Error code for client

  • [Response.error.message]: Error message for client

  • [Response.error.details]: Error details (optional, only for 422 and 429)

    • For 422 (Validation): Array of field-level errors
      • [Response.error.details[].field]: Field name that failed validation
      • [Response.error.details[].message]: Specific error message for the field
    • For 429 (Rate Limit): Object with rate limiting information
      • [Response.error.details.retry_after]: Seconds to wait before retry
      • [Response.error.details.limit]: Rate limit threshold
      • [Response.error.details.remaining]: Remaining requests
      • [Response.error.details.reset]: Reset timestamp

Error code definitions

HTTP status code When to use Examples
400 Bad Request Request has invalid technical format - Malformed JSON (parse error)
- Wrong Content-Type
- Missing request body when required
- Field has wrong data type (string instead of number)
401 Unauthorized Not properly authenticated - Missing or invalid Authorization header
- Token expired / revoked
- Invalid JWT signature
403 Forbidden Authenticated but not authorized - Missing required role / scope
- Attempting to access resource outside allowed boundary (different tenant)
- Blocked by policy (e.g. account locked)
404 Not Found Resource does not exist (or is concealed for security) - ID not found
- Endpoint does not exist (possibly framework-generated 404)
422 Unprocessable Entity Payload is valid JSON but fails validation - Missing required field
- Field violates pattern / format (email, enum, length)
- Other business validation errors (e.g. attempt to cancel an already delivered order)
429 Too Many Requests Exceeded rate limit or quota - More than environment variable value [RATE_LIMIT_PER_IP_PER_MINUTE] requests per minute per IP
- More than environment variable value [RATE_LIMIT_PER_USER_PER_HOUR] requests per hour per authenticated user
- Burst limit: More than environment variable value [RATE_LIMIT_PER_SECOND] requests per second
500 Internal Server Error Unexpected server-side error - Unhandled exception
- System / infrastructure failure (DB, cache, internal service)
- Other uncategorized error

Pagination
GET/pagination{?page,limit}

This document describes:

  • The common standard for handling 2 query parameters: limit (maximum number of elements returned per page) and page (specify the page number to return) used for endpoints that return lists

  • The standard pagination response format for all APIs that return paginated data

Validation

  • [Parameters.page]:

    • Must be a positive integer
    • Minimum value: 1

  • [Parameters.limit]:

    • Must be a positive integer
    • Minimum value: 1
    • Maximum value: Environment variable value [MAX_PAGE_SIZE]

Pagination processing

  • Among records that meet search conditions, only return records with order from (([Parameters.page]-1) * [Parameters.limit])+1 to record number [Parameters.page] * [Parameters.limit]

    • Example: [Parameters.page] = 3 & [Parameters.limit] = 20
      Return records from 41st to 60th

  • If [Parameters.page] is not specified, use default value = 1

  • If [Parameters.limit] is not specified, use default value = Environment variable value [DEFAULT_PAGE_SIZE]

Response processing

  • [Response.data.pagination.total_items]: Total number of records that satisfy the search conditions

  • [Response.data.pagination.page_size] = [Parameters.limit]

    • If [Parameters.limit] is not specified, use default value = Environment variable value [DEFAULT_PAGE_SIZE]

  • [Response.data.pagination.total_pages] = [Response.data.pagination.total_items] / [Response.data.pagination.page_size], rounded up to the nearest whole number

  • [Response.data.pagination.current_page] = [Parameters.page]

    • If [Parameters.page] is not specified, use default value = 1
    • If [Response.data.pagination.total_items] = 0, response 1

Sort
GET/sort{?sort}

This document describes the common standard for handling query parameter sort (specify the order of returned results).

Syntax

  • sort=<field>: Ascending order (default)
    Example: ?sort=name (ascending by name)

  • sort=-<field>: Descending order (minus sign - before field name)
    Example: ?sort=-created_at (descending by creation time)

  • Can sort by one or multiple fields. For multiple fields: separate with comma , Example: ?sort=-created_at,name (descending by creation time, then ascending by name)

Validation

  • [Parameters.sort]:
    • If syntax is incorrect (e.g., sort=,-name), return 422 error SORT_SYNTAX_ERROR
    • Maximum 5 fields allowed per request
      If more than 5 fields specified, return 422 error SORT_TOO_MANY_FIELDS
    • Only allowed to sort by fields in whitelist specified in each API design If field is not in whitelist, return 422 error SORT_FIELD_INVALID

Sort processing

  • If no minus sign - before field name: sort ASC

  • If minus sign - before field name: sort DESC

  • For multiple fields: Priority sort by earlier fields first, later fields second

  • Default behavior: null values at end for ascending, at beginning for descending

  • Stable sort: if field values are equal, original order (or default key like id ascending) is preserved

Filter
GET/filter{?name,category_id,status,in_stock,product_code,price,created_at}

This document describes the common standard for handling query parameters with filtering functionality

Filter processing

  • When the same parameter name receives multiple values, join the conditions with OR

    • Example 1: ?tel=123&tel=456
      Search condition will be
      WHERE (tel LIKE '%123%' OR tel LIKE '%456%')
    • Example 2: ?status=1&status=2
      Search condition will be
      WHERE (status = 1 OR status = 2)

  • For parameters with from-to search functionality that work as a pair, must send/receive in array format as follows

    • Example: ?start_date=[2024-01-01,2024-01-31]&start_date=[2024-03-01,2024-03-31]
      Search condition will be
      WHERE 
    (
        (start_date >= '2024-01-01' AND start_date <= '2024-01-31')
        OR
        (start_date >= '2024-03-01' AND start_date <= '2024-03-31')
    )

  • For parameters that work as a pair, must send/receive in array format as follows

    • Example: ?product_code=[LAP,1203]&product_code=[SP,3201]
      Search condition will be
      WHERE
    (
        (product_category = 'LAP' AND product_id = '1203')
        OR
        (product_category = 'SP' AND product_id = '3201')
    )

  • Join search conditions for different parameter names with AND

    • Example: ?tel=123&tel=456&status=1&status=2&status=3&start_date=[2024-01-01,2024-01-31]&start_date=[2024-03-01,2024-03-31]
      Search condition will be
      WHERE
    (tel LIKE '%123%' OR tel LIKE '%456%')<br />
    AND (status = 1 OR status = 2 OR status = 3)<br />
    AND (
        (start_date >= '2024-01-01' AND start_date <= '2024-01-31')
        OR
        (start_date >= '2024-03-01' AND start_date <= '2024-03-31')
    )
